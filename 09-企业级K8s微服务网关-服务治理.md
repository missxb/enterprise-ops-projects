# 企业级K8s微服务网关 + 服务治理

> 完整实现微服务架构下的网关、服务发现、流量管理、熔断降级、链路追踪

> **⚠️ 版本说明**：本文档基于Kong 3.9.1 + Istio 1.29.2（2026年5月）编写。
> - Kong 3.9.1是当前最新稳定版，支持K8s Gateway API
> - Istio 1.29.2是LTS版本，支持Ambient Mesh（已GA，取消Sidecar，资源消耗降低40-60%）
> - 如需使用Ambient Mesh，请参考Istio官方文档迁移指南
>
> ```yaml
> # Ambient Mesh配置示例(非Sidecar模式)
> # 1. 安装Ambient模式Istio:
> # istioctl install --set profile=ambient
> # 2. 为namespace启用Ambient:
> # kubectl label namespace production istio.io/dataplane-mode=ambient
> # 3. 部署Waypoint Proxy(可选,L7策略需要):
> # kubectl apply -f - <<EOF
> # apiVersion: gateway.networking.k8s.io/v1
> # kind: Gateway
> # metadata:
> #   name: production-waypoint
> #   namespace: production
> #   annotations:
> #     istio.io/waypoint-for: all
> # spec:
> #   gatewayClassName: istio
> # EOF
> ```
>
> Ambient Mesh取消Sidecar,资源消耗降低40-60%。新部署建议使用Ambient模式。

> **⚠️ Kong Manager与DB-less模式不兼容**
>
> Kong Manager是Kong Enterprise版提供的Web管理界面，用于可视化管理路由、服务、插件等配置。
> **DB-less模式下Kong Manager完全不可用**，原因如下：
> - Kong Manager依赖Admin API的CRUD操作来管理配置
> - DB-less模式的Admin API只支持`/config`端点的完整配置替换，不支持增量操作
> - Kong Manager前端会调用`/services`、`/routes`、`/plugins`等端点，这些在DB-less下不存在
>
> **替代方案**:
> 1. **Kong Konnect (SaaS)** - Kong官方SaaS管理平台，支持DB-less模式，推荐生产使用
> 2. **deck** - Kong官方CLI工具，支持声明式配置管理: `deck sync --kong-addr http://kong:8001`
> 3. **Konga** - 开源Web UI，通过文件系统同步配置（社区维护，不再活跃）
> 4. **Kong Admin API + curl/kubectl** - 直接通过`/config`端点上传完整YAML声明式配置
> 5. **升级到Kong Enterprise + PostgreSQL** - 如需完整Kong Manager功能，需使用传统数据库模式
>
> **推荐**: 生产环境使用`deck`工具管理DB-less配置，结合CI/CD流水线实现GitOps:
> ```bash
> # deck同步示例
> deck sync --kong-addr http://kong-admin.kong:8001 \
>   --state-dir ./kong-config/ \
>   --select-tag production
> ```

> DB-less模式不支持Kong Manager(Web UI)。如需管理界面,考虑:
> 1. 使用Kong Manager的Kong Enterprise版
> 2. 使用Kong Admin API + 自定义Dashboard
> 3. 使用Kong的Konnect SaaS管理平台

> **⚠️ DB-less模式 local限流 per-replica计数警告**
>
> `policy: local` 在DB-less模式下是唯一可用的限流策略，但它存在**严重的per-replica计数问题**：
>
> **问题描述**:
> - 每个Kong副本独立维护自己的限流计数器
> - 请求被负载均衡到哪个副本，就由该副本独立计数
> - 副本之间不共享计数状态
>
> **实际影响**（假设配置 `minute: 100`，3个Kong副本）:
> ```
> 理论限流: 100次/分钟
> 实际限流: 100 × 3 = 300次/分钟（最坏情况，请求均匀分布时）
> 最小限流: 100次/分钟（最坏情况，所有请求落到同一个副本时）
> ```
>
> **为什么这是个问题**:
> 1. 负载均衡不均匀时，限流行为不可预测
> 2. 某个副本被驱逐（Pod eviction）后，其计数器丢失，其他副本继续计数
> 3. HPA扩缩容时，副本数变化会直接影响限流阈值
>
> **生产环境应对方案（优先级从高到低）**:
>
> | 方案 | 适用场景 | 说明 |
> |------|---------|------|
> | 1. 启用Kong PostgreSQL + redis策略 | 需要精确全局限流 | 使用Redis共享计数器，最准确 |
> | 2. 使用Redis策略（Kong Enterprise） | 有Redis基础设施 | Redis单机或Sentinel，不支持Cluster |
> | 3. 调低local策略阈值 | 简单场景 | 将阈值设为目标值的 1/副本数，如目标100，设33 |
> | 4. 在Ingress层增加限流 | 外部流量 | 使用Nginx/OpenResty的limit_req做粗粒度限流 |
> | 5. 使用Kubernetes Gateway API | 新项目 | 通过HTTPRoutePolicy实现集群级限流 |
>
> **HPA场景下的特别注意**:
> ```bash
> # 计算实际限流阈值时，需考虑maxReplicas
> # HPA maxReplicas=10, local minute=100
> # 实际最大限流 = 100 × 10 = 1000次/min
> # 如果你只需要100次/min，local策略在HPA下完全不可用
> ```

### 金丝雀自动晋升：基于错误率的Prometheus告警规则

```yaml
# canary-promotion-rules.yaml
# 当canary版本错误率持续低于阈值时，触发晋升告警
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: canary-promotion
  namespace: monitoring
  labels:
    role: canary
spec:
  groups:
  - name: canary.promotion
    rules:
    # canary错误率持续低于1%超过5分钟 → 可以晋升
    - alert: CanaryReadyForPromotion
      expr: |
        (
          sum(rate(istio_requests_total{response_code=~"5..", destination_workload_name=~"canary"}[5m]))
          /
          sum(rate(istio_requests_total{destination_workload_name=~"canary"}[5m]))
        ) < 0.01
      for: 5m
      labels:
        severity: info
        action: promotion
      annotations:
        summary: "Canary版本错误率持续低于1%，建议晋升"
        description: "canary版本在过去5分钟错误率 < 1%，可安全晋升为stable"

    # canary P99延迟持续低于200ms → 可以晋升
    - alert: CanaryLatencyAcceptable
      expr: |
        histogram_quantile(0.99,
          sum(rate(istio_request_duration_milliseconds_bucket{destination_workload_name=~"canary"}[5m]))
          by (le)
        ) < 200
      for: 5m
      labels:
        severity: info
        action: promotion
      annotations:
        summary: "Canary版本P99延迟 < 200ms，建议晋升"
        description: "canary版本在过去5分钟P99延迟 < 200ms，可安全晋升为stable"

    # canary错误率持续高于5%超过2分钟 → 必须回滚
    - alert: CanaryHighErrorRate
      expr: |
        (
          sum(rate(istio_requests_total{response_code=~"5..", destination_workload_name=~"canary"}[5m]))
          /
          sum(rate(istio_requests_total{destination_workload_name=~"canary"}[5m]))
        ) > 0.05
      for: 2m
      labels:
        severity: critical
        action: rollback
      annotations:
        summary: "🚨 Canary版本错误率 > 5%，需立即回滚！"
        description: "canary版本错误率超过5%，已超过安全阈值，必须回滚"

    # canary P99延迟持续高于500ms → 必须回滚
    - alert: CanaryHighLatency
      expr: |
        histogram_quantile(0.99,
          sum(rate(istio_request_duration_milliseconds_bucket{destination_workload_name=~"canary"}[5m]))
          by (le)
        ) > 500
      for: 2m
      labels:
        severity: critical
        action: rollback
      annotations:
        summary: "🚨 Canary版本P99延迟 > 500ms，需立即回滚！"
        description: "canary版本延迟严重超标，必须回滚"
```

### 自动回滚触发条件

| 指标 | 回滚阈值 | 检测窗口 | 动作 |
|------|---------|---------|------|
| 5xx错误率 | > 5% | 连续2分钟 | 立即回滚，将100%流量切回stable |
| P99延迟 | > 500ms | 连续2分钟 | 立即回滚 |
| Pod CrashLoop | 重启 > 3次 | 5分钟内 | 立即回滚 |
| 5xx错误率 | > 1% | 连续5分钟 | 告警，人工确认是否回滚 |

```bash
#!/bin/bash
# canary_rollback.sh - 金丝雀自动回滚脚本
# 可被Prometheus Alertmanager的webhook调用，或配合Argo Events使用

set -euo pipefail

SERVICE=${1:?"用法: $0 <service-name> [namespace]"}
NAMESPACE=${2:-production}
REASON=${3:-"auto-rollback"}

echo "========== 金丝雀回滚: ${SERVICE} =========="
echo "原因: ${REASON}"
echo "时间: $(date '+%Y-%m-%d %H:%M:%S')"

# Step 1: 将VirtualService流量100%切回stable
echo "Step 1: 流量切回stable..."
kubectl apply -f - << EOF
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: ${SERVICE}
  namespace: ${NAMESPACE}
  annotations:
    rollback-reason: "${REASON}"
    rollback-time: "$(date -u +%Y-%m-%dT%H:%M:%SZ)"
spec:
  hosts:
    - ${SERVICE}
  http:
    - route:
        - destination:
            host: ${SERVICE}
            subset: stable
          weight: 100
EOF

# Step 2: 删除canary Deployment
echo "Step 2: 删除canary Deployment..."
kubectl delete deployment ${SERVICE}-canary -n ${NAMESPACE} --ignore-not-found

# Step 3: 删除canary DestinationRule subsets（如有）
echo "Step 3: 清理canary subset..."
kubectl get destinationrule ${SERVICE} -n ${NAMESPACE} -o yaml 2>/dev/null | \
  kubectl replace -f - 2>/dev/null || echo "  DestinationRule不存在或无需修改"

# Step 4: 验证

## 六、网关方案对比与选型

### 6.1 主流网关对比

| 特性 | Kong | Apache APISIX | Nginx Ingress | Envoy |
|------|------|--------------|---------------|-------|
| 基于 | OpenResty | OpenResty | Nginx | C++ |
| 配置方式 | Admin API + DB | etcd + Admin API | K8s CRD | xDS API |
| 插件生态 | 200+ | 80+ | 50+ | 30+ |
| 性能 | 极高 | 极高 | 高 | 高 |
|| 动态配置 | ✅ | ✅ | ❌ 需reload | ✅ |
|| 热更新 | ✅ | ✅ | ⚠️ 部分支持 | ✅ |
|| 服务发现 | ✅ DNS | ✅ DNS/etcd | ✅ K8s | ✅ xDS |
|| 限流 | ✅ 内置 | ✅ 内置 | ✅ 内置 | ✅ 内置 |
```

> **⚠️ Nginx Ingress Controller 动态配置说明**
>
> Nginx Ingress Controller 基于 OpenResty（Nginx + LuaJIT），具备一定的动态配置能力：
> - **动态部分**: Lua-based 功能（如自定义错误页、Lua插件）无需reload即可生效
> - **需要reload的部分**: Ingress规则变更、TLS证书更新、upstream变更等核心配置变更需要执行`nginx -s reload`
> - **性能影响**: reload是平滑的（旧worker处理完已有连接后退出），但高并发下仍可能有短暂的连接重试
>
> **与 Kong/APISIX 的关键区别**:
> | 操作 | Nginx Ingress | Kong | APISIX |
> |------|--------------|------|--------|
> | 新增Ingress规则 | 需reload | 即时生效 | 即时生效 |
> | 修改路由 | 需reload | 即时生效 | 即时生效 |
> | 插件配置变更 | 部分需reload | 即时生效 | 即时生效 |
> | 证书热更新 | 需reload | 即时生效 | 即时生效 |
>
> **生产建议**: 如需频繁动态变更路由/插件配置，推荐使用 Kong 或 APISIX；
> Nginx Ingress 更适合配置变更不频繁的静态路由场景。
> 如确需Nginx Ingress动态能力，可使用 NGINX Ingress Controller 的
> `nginx.ingress.kubernetes.io/configuration-snippet` 注解嵌入Lua代码，
> 或使用 ExternalDNS + cert-manager 实现自动化配置更新。
```
| 运维界面 | ✅ Kong Manager | ✅ Dashboard | ❌ | ❌ |
| 学习曲线 | ⭐⭐ | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| 适用场景 | 通用API网关 | 高性能网关 | K8s Ingress | Service Mesh |

**选型建议**:
- K8s环境 + 需要Ingress → Nginx Ingress Controller
- 需要丰富插件生态 → Kong 或 APISIX
- 微服务架构 + Service Mesh → Envoy (Istio数据面)
- 本项目选择: Nginx Ingress(入口) + Istio(服务间)

### 6.2 服务网格 vs 传统网关

| 特性 | 传统网关(Nginx/Kong) | Service Mesh(Istio/Envoy) |
|------|---------------------|--------------------------|
| 部署方式 | 集中部署 | Sidecar每Pod一个 |
| 管理面 | 网关配置 | 控制面+数据面 |
| 流量管理 | 网关层 | 应用层(7层) |
| 可观测性 | 网关日志 | 全链路追踪+指标 |
| 安全 | TLS终结 | mTLS双向认证 |
| 灰度发布 | 网关路由 | 流量权重分配 |
| 故障注入 | ❌ | ✅ |
| 资源开销 | 低 | 中(每Pod增加~50MB) |
| 适用规模 | 100+服务 | 500+服务 |

> **选型建议**: 小中型集群(<100服务)使用Kong作为API网关即可；大型集群(500+服务)且需mTLS/故障注入/全链路追踪时，引入Istio。两者可共存：Kong管理南北流量(外部→集群)，Istio管理东西流量(服务间)。

### 7.3 mTLS配置

```yaml
# peer-authentication.yaml
---

# 迁移完成后应将所有服务升级为支持mTLS，然后移除PERMISSIVE配置
# 允许非mTLS流量(兼容旧服务，过渡方案)
apiVersion: security.istio.io/v1

```

> **⚠️ PERMISSIVE → STRICT 迁移计划**
>
> 直接将namespace级PeerAuthentication从PERMISSIVE切换到STRICT会导致未注入sidecar的服务
> 全部不可用。必须按以下步骤逐步迁移：
>
> **迁移步骤**:
>
> **阶段1: 盘点（Day 0）**
> ```bash
> # 列出所有未注入sidecar的Pod
> kubectl get pods -n production -o jsonpath='{range .items[?(@.metadata.annotations.sidecar\.istio\.io/inject!="true")]}{.metadata.name}{"\n"}{end}'
> # 检查所有Deployment是否启用了sidecar注入
> kubectl get deployments -n production -o jsonpath='{range .items[*]}{.metadata.name}: annotations={.spec.template.metadata.annotations}{"\n"}{end}'
> # 确认所有ServiceAccount已创建
> kubectl get serviceaccounts -n production
> ```
>
> **阶段2: 逐服务启用mTLS（Day 1-7）**
> ```bash
> # 为每个服务单独设置PERMISSIVE（确保该服务的sidecar已就绪）
> # 在namespace为STRICT之前，逐个服务验证mTLS工作正常

> # 确认证书签发: cipher=TLS_AES_256_GCM_SHA384
> ```
>
> **阶段3: 逐namespace切换到STRICT（Day 7-14）**
> ```bash
> # 1. 确认namespace下所有Pod都有sidecar
> kubectl get pods -n production -o jsonpath='{range .items[*]}{.metadata.name}: {.spec.containers[*].name}{"\n"}{end}' | grep -v istio-proxy

> # 2. 移除所有PERMISSIVE的PeerAuthentication
> kubectl delete peerauthentication legacy-service -n production --ignore-not-found
>
> # 3. 确认namespace级STRICT已生效
> kubectl get peerauthentication -n production -o yaml
>
> # 4. 验证mTLS连接
> istioctl x describe pod <pod-name> -n production
> # 应显示: "Queued for mtls STRICT"

### 7.4 服务间访问控制(AuthorizationPolicy)

```yaml
# authorization-policy.yaml
---
# 默认拒绝所有服务间流量(零信任)
apiVersion: security.istio.io/v1
kind: AuthorizationPolicy
metadata:
  name: deny-all
  namespace: production
spec: {}
```

> **⚠️ AuthorizationPolicy deny-all 模式详解**
>
> `spec: {}` 表示空规则，在Istio中等价于"拒绝所有入站流量"。这是零信任架构的基石：
>
> **工作原理**:
> - 空的AuthorizationPolicy会匹配所有请求，由于没有任何ALLOW规则，所有请求被拒绝
> - 必须显式声明ALLOW规则才能放行特定流量
> - 没有匹配任何AuthorizationPolicy的流量会被Istio默认放行（但有了deny-all后，
>   所有流量都会匹配到此策略，从而被拒绝）
>
> **必须配套的ALLOW规则**:
> ```yaml
> # 示例: 完整的零信任授权策略组合
> ---
> # 1. 全局拒绝所有（命名空间级）
> apiVersion: security.istio.io/v1
> kind: AuthorizationPolicy
> metadata:
>   name: deny-all
>   namespace: production
> spec: {}
>
> ---
> # 2. 允许Ingress Gateway访问所有服务（外部入口）
> apiVersion: security.istio.io/v1
> kind: AuthorizationPolicy
> metadata:
>   name: allow-ingress-gateway
>   namespace: production
> spec:
>   action: ALLOW
>   rules:
>   - from:
>     - source:
>         principals: ["cluster.local/istio-system/ingressgateway-service-account"]
>
> ---
> # 3. 允许Prometheus抓取指标（监控）
> apiVersion: security.istio.io/v1
> kind: AuthorizationPolicy
> metadata:
>   name: allow-prometheus
>   namespace: production
> spec:
>   action: ALLOW
>   rules:
>   - from:
>     - source:
>         principals: ["cluster.local/monitoring/prometheus"]
>     to:
>     - operation:
>         methods: ["GET"]
>         paths: ["/metrics", "/ready"]
> ```
>
> **排查建议**:
> - 应用被拒绝后，查看Envoy access log确认是否匹配了deny-all策略
> - 使用 `istioctl analyze` 检查AuthorizationPolicy配置错误
> - 新增服务前必须先添加对应的ALLOW规则，否则该服务会被完全阻断
>
> **常见遗漏的ALLOW规则**:
> 1. Prometheus/监控系统的抓取请求
> 2. Kubernetes健康检查（liveness/readiness probe）
> 3. Istio控制面的遥测采集
> 4. 跨命名空间的服务调用

```
# 允许frontend调用user-service的GET和POST
apiVersion: security.istio.io/v1
kind: AuthorizationPolicy
metadata:
  name: allow-frontend-to-user
  namespace: production
spec:
  selector:
    matchLabels:
      app: user-service
  action: ALLOW
  rules:
  - from:
    - source:
        principals: ["cluster.local/ns/production/sa/frontend"]
    to:
    - operation:
        methods: ["GET", "POST"]
        paths: ["/api/v1/users/*"]

---
# 允许user-service调用order-service
apiVersion: security.istio.io/v1
kind: AuthorizationPolicy
metadata:
  name: allow-user-to-order
  namespace: production
spec:
  selector:
    matchLabels:
      app: order-service
  action: ALLOW
  rules:
  - from:
    - source:
        principals: ["cluster.local/ns/production/sa/user-service"]
```

### 7.5 Kong + Istio共存架构

> **南北流量**: Kong管理外部→集群的流量(认证、限流、API版本管理)
> **东西流量**: Istio管理集群内部服务间流量(mTLS、熔断、灰度)

```
外部用户 → Kong(API网关, TLS终结) → Ingress → [Istio Mesh: 服务A ↔ 服务B ↔ 服务C]
                                              ↑ mTLS加密 ↑
```

> **⚠️ Kong + Istio mTLS 端到端加密详解**
>
> 在 Kong + Istio 共存架构中，请求经历以下加密阶段：
>
> ```
> 客户端                  Kong Gateway            Istio Mesh
>   │                        │                      │
>   │  1. HTTPS (TLS 1.2+)   │                      │
>   │ ─────────────────────►  │                      │
>   │                        │  TLS终结(Kong侧)      │
>   │                        │  HTTP明文传递给上游    │
>   │                        │  ┌─────────────────┐  │
>   │                        │  │  2. mTLS (Istio) │  │
>   │                        │  │  IngressGateway  │  │
>   │                        │  │  ──────────────► │  │
>   │                        │  │  3. mTLS (Pod间)  │  │
>   │                        │  │  服务A ↔ 服务B   │  │
>   │                        │  └─────────────────┘  │
>   │                        │                      │
> ```
>
> **关键配置点**:
> 1. **Kong → IngressGateway**: Kong作为TLS终结点，将解密后的HTTP流量发送给Istio IngressGateway。
>    如果Kong和IngressGateway在不同Pod，它们之间的通信需要配置：
>    - 方案A: 同Pod部署（sidecar模式），mTLS自动生效
>    - 方案B: 跨Pod部署，在DestinationRule中配置`trafficPolicy.tls.mode: ISTIO_MUTUAL`
>
> 2. **IngressGateway → 应用Pod**: Istio自动为所有注入sidecar的Pod建立mTLS，无需应用代码修改。
>
> 3. **证书管理**: Istio的istiod自动签发和轮换mTLS证书（有效期24小时，默认自动轮换）。
>    Kong使用外部TLS证书（通过cert-manager自动签发Let's Encrypt证书）。
>
> **常见问题**:
> - Kong和IngressGateway之间的HTTP明文流量是否安全？
>   → 如果在同一个K8s集群内（同Node或CNI网络），且网络策略限制了外部访问，则可接受。
>   → 高安全要求场景：在Kong侧也启用到IngressGateway的mTLS（需配置Kong的上游TLS）。
> - Kong的TLS证书和Istio的mTLS证书是独立管理的，不要混淆：
>   - Kong TLS: 面向外部用户，使用Let's Encrypt公共证书
>   - Istio mTLS: 面向内部服务，使用istiod自签发的内部CA证书
>
> **Kong上游mTLS配置示例**（可选，增强Kong→IngressGateway的安全性）:
> ```yaml
> # Kong Service配置 - 使用mTLS连接上游
> apiVersion: configuration.konghq.com/v1
> kind: KongIngress
> metadata:
>   name: kong-proxy
> spec:
>   proxy:
>     protocol: https
>     ssl_verify: false  # Istio使用自签证书，需关闭验证
> ```
```

## 九、真实故障案例

### 案例1: Istio Sidecar导致Pod启动延迟30秒

> **Sidecar资源消耗建议**:
> - CPU: 每个Sidecar约0.1-0.5核(根据流量)
> - 内存: 每个Sidecar约64-128MB
> - 启动延迟: 应用需等待Sidecar就绪(约10-30秒)
> - mTLS性能: 加密/解密约增加5-10%延迟

**故障**: 应用Pod启动后30秒才Ready，影响滚动更新速度

**原因**: Istio sidecar默认等待应用容器完全启动后才注入

**解决**:
```yaml
# 在Pod annotation中添加
metadata:
  annotations:
    proxy.istio.io/config: '{"holdApplicationUntilProxyStarts": true}'
    
# 或者在MeshConfig中全局配置
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    defaultConfig:
      holdApplicationUntilProxyStarts: true
```

### 案例2: 服务网格延迟增加5ms

**故障**: 启用Istio后P99延迟从15ms增加到20ms

**原因**: Sidecar代理增加了网络跳数，遥测采集增加了开销

**解决**:
```yaml
# 1. 降低采样率
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: reduce-sampling
spec:
  tracing:
    - randomSamplingPercentage: 1  # 从10%降到1%

# 2. 禁用不必要的访问日志
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: disable-access-log
spec:
  accessLogging:
    - providers: []  # 禁用

# 3. 使用headless service减少DNS查询
```

### 案例3: Kong限流在Redis集群下失效

**故障**: Kong rate-limiting插件配置了Redis集群，但限流不生效

**原因**: Kong默认使用Redis单机模式连接，不支持Redis Cluster

**解决**:
```yaml
# DB-less模式下(推荐): 直接使用local策略,不依赖任何外部存储
# [注意] DB-less模式不支持cluster/redis策略,local是唯一可用选项
# 每个Kong实例独立计数,适合无状态限流场景
plugins:
  - name: rate-limiting
    config:
      minute: 100
      policy: local  # DB-less模式仅支持local

### 11.2 紧急预案

```bash
#!/bin/bash
# emergency_runbook.sh - K8s微服务网关紧急故障处理脚本
# 用法: ./emergency_runbook.sh <fault-type> [service] [namespace]
#   fault-type: 502 | 504 | OOM | MTLS | CIRCUIT_BREAKER

set -euo pipefail

FAULT_TYPE=${1:?"用法: $0 <502|504|OOM|MTLS|CIRCUIT_BREAKER> [service] [namespace]"}
SERVICE=${2:-"all"}
NAMESPACE=${3:-production}

echo "============================================"
echo "  K8s微服务网关紧急故障处理"
echo "  故障类型: ${FAULT_TYPE}"
echo "  目标服务: ${SERVICE}"
echo "  时间: $(date '+%Y-%m-%d %H:%M:%S')"
echo "============================================"

# Step 1: 快速诊断 - 集群概览
echo ""
echo "[1/5] 集群快速诊断..."
echo "--- 节点状态 ---"
kubectl get nodes -o wide
echo ""
echo "--- 非Running Pod ---"
kubectl get pods -A | grep -v Running | grep -v Completed || echo "  所有Pod正常运行"
echo ""
echo "--- Istio控制面 ---"
kubectl get pods -n istio-system
echo ""
echo "--- Isto分析 ---"
istioctl analyze -n ${NAMESPACE} 2>/dev/null || echo "  istioctl分析完成"

case "${FAULT_TYPE}" in
  502)
    echo ""
    echo "[2/5] 502 Bad Gateway 故障处理..."
    echo "--- 检查后端Pod状态 ---"
    if [ "${SERVICE}" != "all" ]; then
      kubectl get pods -n ${NAMESPACE} -l app=${SERVICE}
    else
      kubectl get pods -n ${NAMESPACE}
    fi
    echo ""
    echo "--- 检查Istio sidecar状态 ---"
    kubectl get pods -n ${NAMESPACE} -o jsonpath='{range .items[*]}{.metadata.name}: {range .status.conditions[*]}{.type}={.status} {" "}{end}{"\n"}{end}'
    echo ""
    echo "--- 最近Envoy错误日志 ---"
    if [ "${SERVICE}" != "all" ]; then
      kubectl logs -n ${NAMESPACE} -l app=${SERVICE} --tail=20 -c istio-proxy 2>/dev/null | grep -i "error\|502" || echo "  未发现错误"
    fi
    echo ""
    echo "--- 检查Service Endpoints ---"
    if [ "${SERVICE}" != "all" ]; then
      kubectl get endpoints ${SERVICE} -n ${NAMESPACE}
    fi
    echo ""
    echo ">>> 处理建议:"
    echo "  1. 确认后端Pod是否Ready"
    echo "  2. 检查Service selector与Pod label是否匹配"
    echo "  3. 检查Envoy sidecar日志"
    echo "  4. 临时方案: kubectl rollout restart deployment/<service> -n ${NAMESPACE}"
    ;;

  504)
    echo ""
    echo "[2/5] 504 Gateway Timeout 故障处理..."
    echo "--- 检查VirtualService超时配置 ---"
    kubectl get virtualservice -n ${NAMESPACE} -o yaml 2>/dev/null | grep -A2 "timeout:" || echo "  未配置timeout"
    echo ""
    echo "--- 检查DestinationRule熔断配置 ---"
    kubectl get destinationrule -n ${NAMESPACE} -o yaml 2>/dev/null | grep -A3 "outlierDetection" || echo "  未配置熔断"
    echo ""
    echo ">>> 处理建议:"
    echo "  1. 检查目标服务日志: kubectl logs -n ${NAMESPACE} -l app=<service>"
    echo "  2. 检查VirtualService timeout是否过短"
    echo "  3. 检查DestinationRule熔断阈值是否过低"
    echo "  4. 临时方案: 增加超时时间或禁用熔断"
    ;;

  OOM)
    echo ""
    echo "[2/5] Pod OOM 故障处理..."
    echo "--- 检查最近OOM事件 ---"
    kubectl get events -n ${NAMESPACE} --field-selector reason=OOMKilling --sort-by='.lastTimestamp' 2>/dev/null | tail -10
    echo ""
    echo "--- 当前Pod内存使用 ---"
    kubectl top pods -n ${NAMESPACE} --sort-by=memory 2>/dev/null || echo "  需要Metrics Server"
    echo ""
    echo ">>> 处理建议:"
    echo "  1. 临时方案: 增加内存limit"
    echo "     kubectl patch deployment <service> -n ${NAMESPACE} --type='json' -p='[{\"op\":\"replace\",\"path\":\"/spec/template/spec/containers/0/resources/limits/memory\",\"value\":\"4Gi\"}]'"
    echo "  2. 检查Envoy内存: kubectl exec -it <pod> -c istio-proxy -- curl -s localhost:15000/stats | grep memory"
    echo "  3. 长期方案: 修复应用内存泄漏"
    ;;

  MTLS)
    echo ""
    echo "[2/5] mTLS通信故障处理..."
    echo "--- 检查PeerAuthentication ---"
    kubectl get peerauthentication -n ${NAMESPACE} -o wide
    echo ""
    echo "--- 检查AuthorizationPolicy ---"
    kubectl get authorizationpolicy -n ${NAMESPACE} -o wide
    echo ""
    echo ">>> 处理建议:"
    echo "  1. 诊断: istioctl analyze -n ${NAMESPACE}"
    echo "  2. 临时方案: 将PeerAuthentication改为PERMISSIVE"
    echo "     kubectl patch peerauthentication default -n ${NAMESPACE} --type='json' -p='[{\"op\":\"replace\",\"path\":\"/spec/mtls/mode\",\"value\":\"PERMISSIVE\"}]'"
    echo "  3. 验证: istioctl x describe pod <pod> -n ${NAMESPACE}"
    ;;

  CIRCUIT_BREAKER)
    echo ""
    echo "[2/5] 熔断器误触发 故障处理..."
    echo "--- 检查熔断器状态 ---"
    kubectl get destinationrule -n ${NAMESPACE} -o yaml 2>/dev/null | grep -A5 "outlierDetection"
    echo ""
    echo "--- 检查Envoy熔断指标 ---"
    if [ "${SERVICE}" != "all" ]; then
      kubectl exec -it $(kubectl get pod -n ${NAMESPACE} -l app=${SERVICE} -o jsonpath='{.items[0].metadata.name}') \
        -c istio-proxy -- curl -s localhost:15000/stats 2>/dev/null | grep "outlier_detection" || echo "  无法获取指标"
    fi
    echo ""
    echo ">>> 处理建议:"
    echo "  1. 临时方案: 降低熔断敏感度"
    echo "  2. 调高consecutive5xxErrors阈值"
    echo "  3. 缩短baseEjectionTime恢复时间"
    ;;

  *)
    echo ""
    echo "未知故障类型: ${FAULT_TYPE}"
    echo "支持的类型: 502, 504, OOM, MTLS, CIRCUIT_BREAKER"
    exit 1
    ;;
esac

echo ""
echo "[3/5] 事件检查..."
kubectl get events -n ${NAMESPACE} --sort-by='.lastTimestamp' --field-selector type=Warning 2>/dev/null | tail -15

echo ""
echo "[4/5] 资源使用概览..."
kubectl top pods -n ${NAMESPACE} --sort-by=cpu 2>/dev/null | head -10 || echo "  需要Metrics Server"

echo ""
echo "[5/5] 完成"
echo "============================================"
echo "故障处理完成，如需进一步排查请检查:"
echo "  - Istio控制面日志: kubectl logs -n istio-system -l app=istiod"
echo "  - Envoy配置: kubectl exec -it <pod> -c istio-proxy -- curl -s localhost:15000/config_dump | jq"
echo "  - istioctl analyze: istioctl analyze -n ${NAMESPACE} --all"
echo "============================================"
```

## 十二、容量规划

> **资源计算公式**:
> - Sidecar开销: 每Pod增加~100MB内存 + 0.1核CPU(Istio Envoy)
> - 总内存 = Σ(服务内存) + (Pod数 × 100MB Sidecar) + 20%预留
> - 总CPU = Σ(服务CPU) + (Pod数 × 0.1核 Sidecar) + 20%预留
> - 存储: Jaeger每100万请求约产生50MB追踪数据(1%采样率)

> **建议**:
|------|------------|---------|
| CPU | 0.1核 | 每Pod预留0.1核 |
| 内存 | 50MB | 每Pod预留128MB |
| 延迟开销 | 1-5ms | 可接受范围 |
| 最大QPS | 10,000+ | 单代理 |

**集群规模建议**:
- 100个服务以下 → 可以不用Service Mesh
- 100-500个服务 → Istio轻量级部署
- 500+服务 → Istio生产级部署+高可用

## 十四、真实故障案例深度分析

### 案例1：Envoy Sidecar OOM导致Pod频繁重启

**故障现象**: 生产环境某个关键服务Pod频繁被OOM Kill，每30分钟重启一次，影响用户下单。

**故障时间线**:
```
10:00  监控告警: order-service Pod重启
10:05  确认Pod被OOM Kill，Exit Code 137
10:10  检查Pod内存使用: Envoy容器占用1.8Gi(限制2Gi)
10:15  检查Envoy内存趋势: 从500MB持续增长到1.8Gi
10:20  排查发现: 大量HTTP/2流未正确关闭
10:25  临时方案: 增加Envoy内存限制到4Gi
10:30  根因修复: 修复应用HTTP/2连接泄漏
```

**根因分析**:
```yaml
# 问题: 应用使用HTTP/2但未正确关闭流
# Envoy作为sidecar需要维护每个HTTP/2流的状态
# 每个未关闭的流占用约2KB内存
# 10000个未关闭流 = 20MB内存泄漏
# 24小时累积 = 480MB/天

# 检查方法:
kubectl exec -it order-service-xxx -c istio-proxy -- \
  curl -s localhost:15000/stats | grep "http2"

# 异常指标:
envoy_cluster_cx_http2_active 10234  # 异常高
```

**修复方案**:
```yaml
# 1. 修复应用代码: 正确关闭HTTP/2流
# 2. 限制HTTP/2最大流数
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: order-service
spec:
  host: order-service
  trafficPolicy:
    connectionPool:
      http:
        h2UpgradePolicy: UPGRADE
        maxRequestsPerConnection: 100
        http1MaxPendingRequests: 100
        http2MaxRequests: 1000
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 30s
```

**经验教训**:
1. Envoy sidecar的内存限制要预留足够空间
2. 监控envoy_cluster_cx_http2_active指标
3. 应用必须正确管理HTTP/2连接生命周期
4. 定期检查Envoy内存使用趋势

### 案例2：服务发现延迟导致新版本不可用

**故障现象**: 新版本部署后，部分用户仍访问旧版本，金丝雀发布流量分配不生效。

**故障时间线**:
```
14:00  部署新版本user-service v2
14:05  更新VirtualService权重(90/10)
14:10  验证发现仍有30%流量到v1
14:15  检查Istiod: 服务注册信息更新延迟
14:20  发现: DNS缓存未刷新，Pod IP未更新
14:25  重启受影响Pod的Envoy sidecar
14:30  流量分配恢复正常
```

**根因分析**:
```bash
# Istiod推送配置到Envoy的延迟分析
# 1. Istiod感知Pod变化: ~5秒
# 2. Envoy配置推送: ~3秒
# 3. Envoy配置生效: ~1秒
# 4. DNS缓存TTL: 默认300秒

# 关键: K8s Service的Endpoints更新有延迟
# 当新Pod启动但Endpoints未更新时
# Envoy仍使用旧的Pod IP列表
```

**修复方案**:
```yaml
# 1. 缩短DNS缓存时间
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: user-service
spec:
  host: user-service
  trafficPolicy:
    connectionPool:
      http:
        maxRequestsPerConnection: 100
    loadBalancer:
      simple: LEAST_REQUEST
      localityLbSetting:
        enabled: true

# 2. 使用PreStop Hook加速Pod终止
lifecycle:
  preStop:
    exec:
      command: ["/bin/sh", "-c", "sleep 5"]

# 3. 配置更短的Endpoints同步周期
# kube-proxy: --sync-period=10s
# Istiod: PILOT_ENABLE_LEGACY_ISTIO_SUBSCRIPTION=true
```

**经验教训**:
1. 金丝雀发布后等待足够时间让配置生效
2. 监控Envoy的config_dump确认配置已同步
3. 使用kubectl rollout status等待部署完成
4. 重要流量切换前验证Endpoints更新

### 案例3：限流规则误杀正常用户

**故障现象**: 大促期间，正常用户频繁收到429 Too Many Requests。

**故障诊断过程**:
```bash
# 1. 检查限流配置
kubectl get kongplugin rate-limiting -o yaml
# config:
#   minute: 100
#   policy: redis
#   redis_host: redis-master

# 2. 分析限流key
# 默认使用IP作为key
# CDN后面的用户共享IP，触发限流

# 3. 检查Redis限流计数
redis-cli -h redis-master GET "rate_limit:{10.10.100.50}"
# 计数: 95/100 (接近上限)

# 4. 确认: 同一CDN节点的100+用户共享限流额度
```

**根因分析**:
```yaml
# 问题1: 限流key使用IP，CDN用户共享
# 问题2: 每分钟100次对单用户合理，但对CDN出口IP不合理
# 问题3: 没有区分业务类型的限流策略

# 正确做法:
# 1. 基于用户ID限流(从JWT中提取)
# 2. 不同API使用不同限流策略
# 3. 大促前调整限流阈值
```

**修复方案**:
```yaml
# 基于用户ID的限流
apiVersion: networking.konghq.com/v1
kind: KongPlugin
metadata:
  name: user-rate-limiting
plugin: rate-limiting
config:
  minute: 200
  policy: redis
  redis_host: redis-master
  limit_by: consumer

---
# 不同API不同限流
apiVersion: networking.konghq.com/v1
kind: KongPlugin
metadata:
  name: api-rate-limiting
plugin: rate-limiting
config:
  minute: 500
  policy: redis
  redis_host: redis-master
  path: /api/v1/products
```

**经验教训**:
1. 限流key要基于业务标识而非IP
2. 大促前需要调整限流策略
3. 监控限流触发次数，设置告警
4. 不同API需要不同的限流策略

### 案例4：TLS证书问题导致服务间mTLS失败

**故障现象**: 新部署的服务无法与其他服务通信，返回连接超时。

**故障时间线**:
```
11:00  部署新服务payment-service
11:05  服务启动成功，但无法调用order-service
11:10  检查Envoy日志: TLS handshake failed
11:15  发现: 新服务的ServiceAccount缺少证书
11:20  检查istiod: 证书签发正常
11:25  排查: PeerAuthentication配置为STRICT
11:30  临时方案: 设置为PERMISSIVE
11:35  修复: 为ServiceAccount配置正确的Role
```

**根因分析**:
```yaml
# 问题: 新服务的ServiceAccount没有正确的RBAC权限
# Istio需要ServiceAccount有权限读取自己的证书
# 但新服务的ClusterRoleBinding配置错误

# 检查方法:
kubectl describe serviceaccount payment-service -n production
kubectl get clusterrolebinding -o yaml | grep payment-service

# 错误配置:
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: payment-service
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: istio-cert-manager
subjects:
  - kind: ServiceAccount
    name: payment-service
    namespace: production
```

**修复方案**:
```yaml
# 1. 修正RBAC配置
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: payment-service-cert
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: istiod-cert-manager
subjects:
  - kind: ServiceAccount
    name: payment-service
    namespace: production

# 2. 重启Pod触发证书重新签发
kubectl rollout restart deployment payment-service -n production

# 3. 验证mTLS状态
istioctl x describe pod payment-service-xxx -n production
```

**经验教训**:
1. 新服务部署前检查ServiceAccount权限
2. 使用istioctl analyze检查配置问题
3. mTLS配置变更需要充分测试

### 案例5：熔断器误触发导致服务雪崩

**故障现象**: 一个下游服务超时，导致上游服务熔断器开启，进而影响整个调用链。

**故障时间线**:
```
09:00  库存服务响应变慢(2秒超时)
09:01  订单服务熔断器开启(连续5次5xx)
09:02  用户服务调用订单服务被熔断
09:03  购物车服务调用用户服务被熔断
09:04  级联故障，整个下单链路不可用
```

**根因分析**:
```yaml
# 熔断器配置过于激进
outlierDetection:
  consecutive5xxErrors: 5
  interval: 30s
  baseEjectionTime: 30s
  maxEjectionPercent: 50

# 问题分析:
# 1. consecutive5xxErrors太低，容易误触发
# 2. interval太长，熔断恢复慢
# 3. 没有考虑不同服务的特性差异
```

**修复方案**:
```yaml
# 差异化熔断配置
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: inventory-service
spec:
  host: inventory-service
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
      http:
        http1MaxPendingRequests: 50
        http2MaxRequests: 500
    outlierDetection:
      consecutive5xxErrors: 10
      interval: 10s
      baseEjectionTime: 30s
      maxEjectionPercent: 30
      minHealthPercent: 50

---
# 关键服务: 更宽松的熔断策略
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: order-service
spec:
  host: order-service
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 200
      http:
        http1MaxPendingRequests: 200
        http2MaxRequests: 2000
    outlierDetection:
      consecutive5xxErrors: 20
      interval: 10s
      baseEjectionTime: 15s
      maxEjectionPercent: 20
```

**经验教训**:
1. 不同服务需要差异化的熔断策略
2. 关键服务要更宽松的熔断配置
3. 监控熔断器状态，设置告警
4. 定期review熔断策略的合理性

### 案例6：CORS配置错误导致前端无法调用API

**故障现象**: 前端部署后，浏览器控制台显示CORS错误，API请求被阻止。

**故障诊断过程**:
```bash
# 1. 浏览器报错
# Access to XMLHttpRequest at 'https://api.ecommerce.com/v1/users' 
# from origin 'https://www.ecommerce.com' 
# has been blocked by CORS policy

# 2. 检查Kong CORS插件配置
kubectl get kongplugin cors -o yaml
# config:
#   origins:
#     - "*"  # 通配符不支持携带凭证

# 3. 分析: 前端使用了withCredentials: true
# 浏览器要求CORS必须指定具体origin，不能用"*"
```

**根因分析**:
```yaml
# 错误配置: 使用"*"作为origin
plugins:
  - name: cors
    config:
      origins:
        - "*"

# 正确配置: 指定具体origin
plugins:
  - name: cors
    config:
      origins:
        - "https://www.ecommerce.com"
        - "https://m.ecommerce.com"
      methods:
        - GET
        - POST
        - PUT
        - DELETE
      headers:
        - Authorization
        - Content-Type
        - X-Request-Id
      credentials: true
      max_age: 3600
```

**修复方案**:
```yaml
# 1. 更新CORS插件配置
apiVersion: networking.konghq.com/v1
kind: KongPlugin
metadata:
  name: cors-production
plugin: cors
config:
  origins:
    - "https://www.ecommerce.com"
    - "https://m.ecommerce.com"
  methods:
    - GET
    - POST
    - PUT
    - DELETE
    - OPTIONS
  headers:
    - Authorization
    - Content-Type
    - X-Request-Id
  credentials: true
  max_age: 3600
  preflight_continue: false

# 2. 使用正则匹配动态origin
origins:
  - "https://*.ecommerce.com"

# 3. 测试验证
curl -H "Origin: https://www.ecommerce.com" \
  -H "Access-Control-Request-Method: POST" \
  -X OPTIONS \
  https://api.ecommerce.com/v1/users
```

**经验教训**:
1. CORS配置要精确指定origin
2. 携带凭证时不能使用"*"
3. 测试环境和生产环境配置要一致
4. 监控CORS相关错误日志

### 案例7：请求体过大导致502错误

**故障现象**: 用户上传大文件时，网关返回502 Bad Gateway。

**故障诊断过程**:
```bash
# 1. 检查Envoy日志
kubectl logs order-service-xxx -c istio-proxy | grep "502"
# [2024-01-15 10:00:00] POST /api/v1/upload 502 upstream error

# 2. 检查Envoy配置
kubectl exec -it order-service-xxx -c istio-proxy -- \
  curl -s localhost:15000/config_dump | jq '.configs[0].dynamic_listeners[0].filter_chains[0].filters[0].typed_config'

# 3. 发现: Envoy默认限制请求体大小为1MB
# 超过限制后Envoy直接返回502
```

**根因分析**:
```yaml
# 问题: Envoy默认配置
# - max_request_bytes: 1048576 (1MB)
# - max_request_header_size: 8192 (8KB)

# 当请求体超过限制时:
# 1. Envoy拒绝接收请求体
# 2. 返回502 Bad Gateway
# 3. 应用层看不到任何请求
```

**修复方案**:
```yaml
# 1. 调整Envoy监听器配置
apiVersion: networking.istio.io/v1
kind: EnvoyFilter
metadata:
  name: upload-size-limit
  namespace: production
spec:
  workloadSelector:
    labels:
      app: order-service
  configPatches:
    - applyTo: HTTP_FILTER
      match:
        context: SIDECAR_INBOUND
        listener:
          filterChain:
            filter:
              name: envoy.filters.network.http_connection_manager
              subFilter:
                name: envoy.filters.http.router
      patch:
        operation: INSERT_BEFORE
        value:
          name: envoy.filters.http.buffer
          typed_config:
            "@type": type.googleapis.com/envoy.extensions.filters.http.buffer.v3.Buffer
            max_request_bytes: 104857600

# 2. 使用独立的上传服务
# 大文件上传不经过Envoy sidecar
# 使用NodePort直接访问应用

# 3. 采用分片上传方案
# 前端将大文件分片
# 每个分片小于Envoy限制
```

**经验教训**:
1. Envoy有默认的请求体大小限制
2. 大文件上传需要特殊处理
3. 监控502错误，及时发现限制问题
4. 考虑使用对象存储直传方案

### 案例8：JWT时钟偏差导致认证失败

**故障现象**: 部分用户登录后，API请求返回401 Unauthorized，但Token未过期。

**故障诊断过程**:
```bash
# 1. 检查Token状态
JWT_TOKEN="eyJhbGciOiJSUzI1NiIs..."
echo $JWT_TOKEN | cut -d. -f2 | base64 -d | jq .
# {
#   "exp": 1705305600,
#   "iat": 1705219200,
#   "sub": "user123"
# }

# 2. 检查当前时间
date +%s
# 1705305590

# 3. 检查Kong JWT插件配置
kubectl get kongplugin jwt -o yaml
# config:
#   claims_to_verify:
#     - exp
#     - iat

# 4. 发现: Kong验证iat时使用服务器时间
# 但服务器时间比NTP服务器快30秒
# 导致Token被认为"未来签发"
```

**根因分析**:
```yaml
# 问题: 服务器时钟与NTP服务器偏差30秒
# JWT验证时:
# 1. 检查exp: 当前时间 < exp ✅
# 2. 检查iat: 当前时间 > iat ✅
# 3. 但Kong内部使用了不同的时间源
# 导致验证失败

# 根本原因:
# - 服务器NTP同步延迟
# - Kong使用Lua的os.time()而非系统时间
# - 时钟偏差超过JWT的允许范围
```

**修复方案**:
```yaml
# 1. 确保NTP同步正常
# 在Pod中配置NTP
# 或使用hostNetwork共享宿主机时间

# 2. Kong JWT插件配置容错
apiVersion: networking.konghq.com/v1
kind: KongPlugin
metadata:
  name: jwt-production
plugin: jwt
config:
  key_claim_name: kid
  claims_to_verify:
    - exp
  maximum_clock_skew: 60
# > 300秒(5分钟)过于宽松,建议60秒。NTP同步良好的环境可使用30秒。

# 3. 增加Token过期缓冲时间
# Token有效期设为2小时，但实际使用1.5小时
# 避免临界时间问题
```

**经验教训**:
1. 所有服务器必须配置NTP同步
2. JWT验证要考虑时钟偏差
3. 监控时钟偏差指标
4. Token有效期要留足缓冲时间

### 19.4 版本升级SOP

#### 方式一：直接升级（简单但不可中断）

```bash
#!/bin/bash
# upgrade_sop.sh - Istio版本升级SOP

set -euo pipefail

echo "========== Istio版本升级 =========="
echo "当前版本: $(istioctl version --remote=false)"
echo "目标版本: $1"

# Step 1: 备份当前配置
echo "Step 1: 备份当前配置"
# [注意] istioctl manifest dump 在Istio 1.29中已废弃
# 必须使用 istioctl install --dry-run 替代:
#   旧: istioctl manifest dump > /tmp/istio_backup.yaml  (已废弃)
#   新: istioctl install --dry-run > /tmp/istio_backup.yaml
istioctl install --dry-run > /tmp/istio_backup.yaml

# Step 2: 下载新版本
echo "Step 2: 下载新版本"
curl -L https://istio.io/downloadIstio | ISTIO_VERSION=$1 sh -

# Step 3: 升级控制面
echo "Step 3: 升级控制面"
istioctl upgrade -f /tmp/istio_backup.yaml -y

# Step 4: 升级数据面
echo "Step 4: 升级数据面(灰度)"
kubectl rollout restart deployment/istio-ingressgateway -n istio-system

# Step 5: 验证
echo "Step 5: 验证升级"
istioctl verify-install
kubectl get pods -n istio-system

# Step 6: 回滚脚本
echo "Step 6: 如需回滚执行:"
echo "  istioctl manifest apply -f /tmp/istio_backup.yaml"

echo "========== 升级完成 =========="
```

#### 方式二：金丝雀升级（推荐生产使用，可无损回滚）

> 使用 `--revision=canary` 可以同时运行新旧两个版本的控制面，通过标签逐步迁移namespace，实现零停机升级。

```bash
#!/bin/bash
# canary_istio_upgrade.sh - Istio金丝雀升级SOP
# 原理: 安装新版控制面(revision=canary) → 逐步迁移namespace → 验证 → 卸载旧版

set -euo pipefail

NEW_VERSION=$1:? "用法: $0 <new-istio-version> [namespace]"
NAMESPACE=${2:-production}
OLD_REVISION=$(kubectl get istiooperator -n istio-system -o jsonpath='{.items[0].spec.revision}' 2>/dev/null || echo "")

echo "============================================"
echo "  Istio金丝雀升级"
echo "  新版本: ${NEW_VERSION}"
echo "  目标namespace: ${NAMESPACE}"
echo "  旧revision: ${OLD_REVISION:-default}"
echo "  时间: $(date '+%Y-%m-%d %H:%M:%S')"
echo "============================================"

# Step 1: 下载新版istioctl
echo "Step 1: 下载新版istioctl..."
curl -L https://istio.io/downloadIstio | ISTIO_VERSION=${NEW_VERSION} sh -
export PATH="$PWD/istio-${NEW_VERSION}/bin:$PATH"

# Step 2: 安装新版控制面（带canary revision标签）
echo "Step 2: 安装新版控制面..."
istioctl install --set revision=canary -y

# Step 3: 确认两个版本的控制面都在运行
echo "Step 3: 确认双版本运行..."
kubectl get pods -n istio-system | grep istiod
# 应该看到: istiod-xxxx (旧版) + istiod-canary-xxxx (新版)

# Step 4: 查看哪些namespace使用旧版本
echo "Step 4: 当前namespace sidecar注入状态..."
kubectl get namespace -o json | \
  jq -r '.items[] | select(.metadata.labels["istio.io/rev"] != null) | "\(.metadata.name)\t\(.metadata.labels["istio.io/rev"])"'

# Step 5: 迁移目标namespace到新版本
echo "Step 5: 迁移 ${NAMESPACE} 到 canary..."
kubectl label namespace ${NAMESPACE} istio.io/rev=canary --overwrite

# Step 6: 触发Pod滚动重启（新sidecar注入）
echo "Step 6: 滚动重启 ${NAMESPACE} 的所有Pod..."
kubectl rollout restart deployment -n ${NAMESPACE}
kubectl rollout status deployment -n ${NAMESPACE} --timeout=600s

# Step 7: 验证新版本
echo "Step 7: 验证新版本..."
for pod in $(kubectl get pods -n ${NAMESPACE} -o jsonpath='{.items[*].metadata.name}'); do
  echo "  ${pod}:"
  kubectl exec -it ${pod} -n ${NAMESPACE} -c istio-proxy -- \
    pilot-agent request GET server_info 2>/dev/null | grep -E "version|istio" || echo "    (无法获取)"
done

# Step 8: 运行诊断
echo "Step 8: 运行诊断..."
istioctl analyze -n ${NAMESPACE}

echo ""
echo "============================================"
echo "  金丝雀升级完成！"
echo "============================================"
echo ""
echo "验证无误后，执行清理步骤:"
echo "  1. 确认所有namespace已迁移: kubectl get namespace -L istio.io/rev"
echo "  2. 卸载旧版控制面: istioctl uninstall --purge"
echo "  3. 清理旧版标签: kubectl label namespace <ns> istio.io/rev-"
echo ""
echo "如需回滚（迁移namespace回旧版本）:"
echo "  kubectl label namespace ${NAMESPACE} istio.io/rev=${OLD_REVISION:-default} --overwrite"
echo "  kubectl rollout restart deployment -n ${NAMESPACE}"
```

**金丝雀升级 vs 直接升级对比**:

| 特性 | 直接升级 | 金丝雀升级（--revision=canary） |
|------|---------|-------------------------------|
| 停机时间 | 有（控制面重启期间） | 无（双版本并行运行） |
| 回滚速度 | 慢（需重新安装旧版） | 快（切回旧revision即可） |
| 复杂度 | 低 | 中 |
| 适用场景 | 测试/开发环境 | 生产环境 |
| namespace迁移 | 一次性 | 逐个namespace迁移 |
