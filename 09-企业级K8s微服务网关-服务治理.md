## 一、架构

```
用户请求 ──▶ Ingress ──▶ API Gateway ──▶ 微服务A ──▶ 微服务B
                  │         │              │
                  │    ┌────▼────┐    ┌────▼────┐
                  │    │ 限流认证 │    │ 熔断降级 │
                  │    │ 路由转发 │    │ 负载均衡 │
                  │    │ 协议转换 │    │ 重试超时 │
                  │    └─────────┘    └─────────┘
                  │
                  ▼
            链路追踪(Jaeger)
            指标监控(Prometheus)
            日志收集(Filebeat)
```

## 六、网关方案对比与选型

### 6.1 主流网关对比

| 特性 | Kong | Apache APISIX | Nginx Ingress | Envoy |
|------|------|--------------|---------------|-------|
| 基于 | OpenResty | OpenResty | Nginx | C++ |
| 配置方式 | Admin API + DB | etcd + Admin API | K8s CRD | xDS API |
| 插件生态 | 200+ | 80+ | 50+ | 30+ |
| 性能 | 极高 | 极高 | 高 | 高 |
| 动态配置 | ✅ | ✅ | ❌ 需reload | ✅ |
| 热更新 | ✅ | ✅ | ❌ | ✅ |
| 服务发现 | ✅ DNS | ✅ DNS/etcd | ✅ K8s | ✅ xDS |
| 限流 | ✅ 内置 | ✅ 内置 | ✅ 内置 | ✅ 内置 |
| 认证 | ✅ JWT/OAuth2 | ✅ JWT/key-auth | ✅ OAuth2 | ✅ JWT |
| 运维界面 | ✅ Kong Manager | ✅ Dashboard | ❌ | ❌ |
| 学习曲线 | ⭐⭐ | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| 适用场景 | 通用API网关 | 高性能网关 | K8s Ingress | Service Mesh |

**选型建议**:
- K8s环境 + 需要Ingress → Nginx Ingress Controller
- 需要丰富插件生态 → Kong 或 APISIX
- 微服务架构 + Service Mesh → Envoy (Istio数据面)
- 本项目选择: Nginx Ingress(入口) + Istio(服务间)

### 6.2 服务网格 vs 传统网关

| 特性 | 传统网关(Nginx/Kong) | Service Mesh(Istio/Envoy) |
|------|---------------------|--------------------------|
| 部署方式 | 集中部署 | Sidecar每Pod一个 |
| 管理面 | 网关配置 | 控制面+数据面 |
| 流量管理 | 网关层 | 应用层(7层) |
| 可观测性 | 网关日志 | 全链路追踪+指标 |
| 安全 | TLS终结 | mTLS双向认证 |
| 灰度发布 | 网关路由 | 流量权重分配 |
| 故障注入 | ❌ | ✅ |
| 资源开销 | 低 | 中(每Pod增加~50MB) |
| 适用规模 | 100+服务 | 500+服务 |

> **选型建议**: 小中型集群(<100服务)使用Kong作为API网关即可；大型集群(500+服务)且需mTLS/故障注入/全链路追踪时，引入Istio。两者可共存：Kong管理南北流量(外部→集群)，Istio管理东西流量(服务间)。

## 九、真实故障案例

### 案例1: Istio Sidecar导致Pod启动延迟30秒

**故障**: 应用Pod启动后30秒才Ready，影响滚动更新速度

**原因**: Istio sidecar默认等待应用容器完全启动后才注入

**解决**:
```yaml
# 在Pod annotation中添加
metadata:
  annotations:
    proxy.istio.io/config: '{"holdApplicationUntilProxyStarts": true}'
    
# 或者在MeshConfig中全局配置
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    defaultConfig:
      holdApplicationUntilProxyStarts: true
```

### 案例2: 服务网格延迟增加5ms

**故障**: 启用Istio后P99延迟从15ms增加到20ms

**原因**: Sidecar代理增加了网络跳数，遥测采集增加了开销

**解决**:
```yaml
# 1. 降低采样率
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: reduce-sampling
spec:
  tracing:
    - randomSamplingPercentage: 1  # 从10%降到1%

# 2. 禁用不必要的访问日志
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: disable-access-log
spec:
  accessLogging:
    - providers: []  # 禁用

# 3. 使用headless service减少DNS查询
```

### 案例3: Kong限流在Redis集群下失效

**故障**: Kong rate-limiting插件配置了Redis集群，但限流不生效

**原因**: Kong默认使用Redis单机模式连接，不支持Redis Cluster

**解决**:
```yaml
# 方案1: 使用Redis Sentinel
plugins:
  - name: rate-limiting
    config:
      minute: 100
      policy: redis
      redis_host: redis-sentinel
      redis_port: 26379
      redis_sentinel_master: mymaster

# 方案2: 使用外部Redis单机(推荐)
# 部署独立的Redis实例给Kong使用
```

## 十、性能压测

### 10.1 k6压测脚本

```javascript
// load-test.js - k6负载测试
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '2m', target: 100 },   // 爬坡
    { duration: '5m', target: 100 },   // 持续负载
    { duration: '2m', target: 500 },   // 峰值
    { duration: '5m', target: 500 },   // 峰值持续
    { duration: '2m', target: 0 },     // 缓降
  ],
  thresholds: {
    http_req_duration: ['p(99)<200'],  // P99 < 200ms
    http_req_failed: ['rate<0.01'],    // 错误率 < 1%
  },
};

const BASE_URL = 'http://istio-ingressgateway:80';

export default function () {
  // 用户列表API
  const usersRes = http.get(`${BASE_URL}/api/v1/users?page=1&size=20`, {
    headers: { Authorization: `Bearer ${__ENV.TOKEN}` },
  });
  check(usersRes, {
    'users status 200': (r) => r.status === 200,
    'users latency < 100ms': (r) => r.timings.duration < 100,
  });

  sleep(1);

  // 创建订单API
  const orderRes = http.post(`${BASE_URL}/api/v1/orders`,
    JSON.stringify({ product_id: 1, quantity: 1 }),
    { headers: { 'Content-Type': 'application/json', Authorization: `Bearer ${__ENV.TOKEN}` } }
  );
  check(orderRes, {
    'order status 201': (r) => r.status === 201,
    'order latency < 200ms': (r) => r.timings.duration < 200,
  });

  sleep(Math.random() * 3 + 1);
}
```

## 十一、运维SOP

### 11.1 金丝雀发布流程

```bash
#!/bin/bash
# canary_release.sh - 金丝雀发布

set -euo pipefail

SERVICE=$1
NAMESPACE=${2:-production}
CANARY_WEIGHT=${3:-10}

echo "========== 金丝雀发布: ${SERVICE} =========="

echo "Step 1: 部署canary版本..."
kubectl apply -f - << EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ${SERVICE}-canary
  namespace: ${NAMESPACE}
spec:
  replicas: 1
  selector:
    matchLabels:
      app: ${SERVICE}
      version: canary
  template:
    metadata:
      labels:
        app: ${SERVICE}
        version: canary
    spec:
      containers:
        - name: ${SERVICE}
          image: harbor.internal.com/production/${SERVICE}:canary
EOF

echo "Step 2: 配置流量分配(${CANARY_WEIGHT}%到canary)..."
kubectl apply -f - << EOF
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: ${SERVICE}
  namespace: ${NAMESPACE}
spec:
  hosts:
    - ${SERVICE}
  http:
    - route:
        - destination:
            host: ${SERVICE}
            subset: stable
          weight: $((100 - CANARY_WEIGHT))
        - destination:
            host: ${SERVICE}
            subset: canary
          weight: ${CANARY_WEIGHT}
EOF

echo "Step 3: 监控canary指标(5分钟)..."
echo "  查看Grafana dashboard: http://grafana.internal.com/d/canary"
echo "  关注: 错误率、延迟、QPS"

echo ""
```

### 案例1：Envoy Sidecar OOM导致Pod频繁重启

**故障现象**: 生产环境某个关键服务Pod频繁被OOM Kill，每30分钟重启一次，影响用户下单。

**故障时间线**:
```
10:00  监控告警: order-service Pod重启
10:05  确认Pod被OOM Kill，Exit Code 137
10:10  检查Pod内存使用: Envoy容器占用1.8Gi(限制2Gi)
10:15  检查Envoy内存趋势: 从500MB持续增长到1.8Gi
10:20  排查发现: 大量HTTP/2流未正确关闭
10:25  临时方案: 增加Envoy内存限制到4Gi
10:30  根因修复: 修复应用HTTP/2连接泄漏
```

**根因分析**:
```yaml
# 问题: 应用使用HTTP/2但未正确关闭流
# Envoy作为sidecar需要维护每个HTTP/2流的状态
# 每个未关闭的流占用约2KB内存
# 10000个未关闭流 = 20MB内存泄漏
# 24小时累积 = 480MB/天

# 检查方法:
kubectl exec -it order-service-xxx -c istio-proxy -- \
  curl -s localhost:15000/stats | grep "http2"

# 异常指标:
envoy_cluster_cx_http2_active 10234  # 异常高
```

**修复方案**:
```yaml
# 1. 修复应用代码: 正确关闭HTTP/2流
# 2. 限制HTTP/2最大流数
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: order-service
spec:
  host: order-service
  trafficPolicy:
    connectionPool:
      http:
        h2UpgradePolicy: DEFAULT
        maxRequestsPerConnection: 100
        http1MaxPendingRequests: 100
        http2MaxRequests: 1000
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 30s
```

**经验教训**:
1. Envoy sidecar的内存限制要预留足够空间
2. 监控envoy_cluster_cx_http2_active指标
3. 应用必须正确管理HTTP/2连接生命周期
4. 定期检查Envoy内存使用趋势

### 案例2：服务发现延迟导致新版本不可用

**故障现象**: 新版本部署后，部分用户仍访问旧版本，金丝雀发布流量分配不生效。

**故障时间线**:
```
14:00  部署新版本user-service v2
14:05  更新VirtualService权重(90/10)
14:10  验证发现仍有30%流量到v1
14:15  检查Istiod: 服务注册信息更新延迟
14:20  发现: DNS缓存未刷新，Pod IP未更新
14:25  重启受影响Pod的Envoy sidecar
14:30  流量分配恢复正常
```

**根因分析**:
```bash
# Istiod推送配置到Envoy的延迟分析
# 1. Istiod感知Pod变化: ~5秒
# 2. Envoy配置推送: ~3秒
# 3. Envoy配置生效: ~1秒
# 4. DNS缓存TTL: 默认300秒

# 关键: K8s Service的Endpoints更新有延迟
# 当新Pod启动但Endpoints未更新时
# Envoy仍使用旧的Pod IP列表
```

**修复方案**:
```yaml
# 1. 缩短DNS缓存时间
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: user-service
spec:
  host: user-service
  trafficPolicy:
    connectionPool:
      http:
        maxRequestsPerConnection: 100
    loadBalancer:
      simple: LEAST_REQUEST
      localityLbSetting:
        enabled: true

# 2. 使用PreStop Hook加速Pod终止
lifecycle:
  preStop:
    exec:
      command: ["/bin/sh", "-c", "sleep 5"]

# 3. 配置更短的Endpoints同步周期
# kube-proxy: --sync-period=10s
# Istiod: PILOT_ENABLE_LEGACY_ISTIO_SUBSCRIPTION=true
```

**经验教训**:
1. 金丝雀发布后等待足够时间让配置生效
2. 监控Envoy的config_dump确认配置已同步
3. 使用kubectl rollout status等待部署完成
4. 重要流量切换前验证Endpoints更新

### 案例3：限流规则误杀正常用户

**故障现象**: 大促期间，正常用户频繁收到429 Too Many Requests。

**故障诊断过程**:
```bash
# 1. 检查限流配置
kubectl get kongplugin rate-limiting -o yaml
# config:
#   minute: 100
#   policy: redis
#   redis_host: redis-master

# 2. 分析限流key
# 默认使用IP作为key
# CDN后面的用户共享IP，触发限流

# 3. 检查Redis限流计数
redis-cli -h redis-master GET "rate_limit:{10.10.100.50}"
# 计数: 95/100 (接近上限)

# 4. 确认: 同一CDN节点的100+用户共享限流额度
```

**根因分析**:
```yaml
# 问题1: 限流key使用IP，CDN用户共享
# 问题2: 每分钟100次对单用户合理，但对CDN出口IP不合理
# 问题3: 没有区分业务类型的限流策略

# 正确做法:
# 1. 基于用户ID限流(从JWT中提取)
# 2. 不同API使用不同限流策略
# 3. 大促前调整限流阈值
```

**修复方案**:
```yaml
# 基于用户ID的限流
apiVersion: networking.konghq.com/v1
kind: KongPlugin
metadata:
  name: user-rate-limiting
plugin: rate-limiting
config:
  minute: 200
  policy: redis
  redis_host: redis-master
  limit_by: consumer

---
# 不同API不同限流
apiVersion: networking.konghq.com/v1
kind: KongPlugin
metadata:
  name: api-rate-limiting
plugin: rate-limiting
config:
  minute: 500
  policy: redis
  redis_host: redis-master
  path: /api/v1/products
  minute: 1000
```

**经验教训**:
1. 限流key要基于业务标识而非IP
2. 大促前需要调整限流策略
3. 监控限流触发次数，设置告警
4. 不同API需要不同的限流策略

### 案例4：TLS证书问题导致服务间mTLS失败

**故障现象**: 新部署的服务无法与其他服务通信，返回连接超时。

**经验教训**:
1. 新服务部署前检查ServiceAccount权限

```bash
# 1. 浏览器报错
# Access to XMLHttpRequest at 'https://api.ecommerce.com/v1/users'
```

```yaml
# 错误配置: 使用"*"作为origin
plugins:
  - name: cors

plugins:
  - name: cors
    config:
      origins:

        - GET
        - POST
        - PUT
        - DELETE
      headers:

        - Content-Type
        - X-Request-Id

      max_age: 3600
```

```yaml
apiVersion: networking.konghq.com/v1

# 3. 测试验证
```

### 案例7：请求体过大导致502错误

**故障现象**: 用户上传大文件时，网关返回502 Bad Gateway。

**故障诊断过程**:
```bash
# 1. 检查Envoy日志
kubectl logs order-service-xxx -c istio-proxy | grep "502"
# [2024-01-15 10:00:00] POST /api/v1/upload 502 upstream error

# 2. 检查Envoy配置
kubectl exec -it order-service-xxx -c istio-proxy -- \
  curl -s localhost:15000/config_dump | jq '.configs[0].dynamic_listeners[0].filter_chains[0].filters[0].typed_config'

# 3. 发现: Envoy默认限制请求体大小为1MB
# 超过限制后Envoy直接返回502
```

**根因分析**:
```yaml
# 问题: Envoy默认配置
# - max_request_bytes: 1048576 (1MB)
# - max_request_header_size: 8192 (8KB)

# 当请求体超过限制时:
# 1. Envoy拒绝接收请求体
# 2. 返回502 Bad Gateway
# 3. 应用层看不到任何请求
```

**修复方案**:
```yaml
# 1. 调整Envoy监听器配置
apiVersion: networking.istio.io/v1beta1
kind: EnvoyFilter
metadata:
  name: upload-size-limit
  namespace: production
spec:
  workloadSelector:
    labels:
      app: order-service
  configPatches:
    - applyTo: HTTP_FILTER
      match:
        context: SIDECAR_INBOUND
        listener:
          filterChain:
            filter:
              name: envoy.filters.network.http_connection_manager
              subFilter:
                name: envoy.filters.http.router
      patch:
        operation: INSERT_BEFORE
        value:
          name: envoy.filters.http.buffer
          typed_config:
            "@type": type.googleapis.com/envoy.extensions.filters.http.buffer.v3.Buffer
            max_request_bytes: 104857600

# 2. 使用独立的上传服务
# 大文件上传不经过Envoy sidecar
# 使用NodePort直接访问应用

# 3. 采用分片上传方案
# 前端将大文件分片
# 每个分片小于Envoy限制
```

**经验教训**:
1. Envoy有默认的请求体大小限制
2. 大文件上传需要特殊处理
3. 监控502错误，及时发现限制问题
4. 考虑使用对象存储直传方案

### 案例8：JWT时钟偏差导致认证失败

**故障现象**: 部分用户登录后，API请求返回401 Unauthorized，但Token未过期。

**故障诊断过程**:
```bash
# 1. 检查Token状态
JWT_TOKEN="eyJhbGciOiJSUzI1NiIs..."
echo $JWT_TOKEN | cut -d. -f2 | base64 -d | jq .
# {
#   "exp": 1705305600,
#   "iat": 1705219200,
#   "sub": "user123"
# }

# 2. 检查当前时间
date +%s
# 1705305590

# 3. 检查Kong JWT插件配置
kubectl get kongplugin jwt -o yaml
# config:
#   claims_to_verify:
#     - exp
#     - iat

# 4. 发现: Kong验证iat时使用服务器时间
# 但服务器时间比NTP服务器快30秒
# 导致Token被认为"未来签发"
```

**根因分析**:
```yaml
# 问题: 服务器时钟与NTP服务器偏差30秒
# JWT验证时:
# 1. 检查exp: 当前时间 < exp ✅
# 2. 检查iat: 当前时间 > iat ✅
# 3. 但Kong内部使用了不同的时间源
# 导致验证失败

# 根本原因:
# - 服务器NTP同步延迟
# - Kong使用Lua的os.time()而非系统时间
# - 时钟偏差超过JWT的允许范围
```

**修复方案**:
```yaml
# 1. 确保NTP同步正常
# 在Pod中配置NTP
# 或使用hostNetwork共享宿主机时间

# 2. Kong JWT插件配置容错
apiVersion: networking.konghq.com/v1
kind: KongPlugin
metadata:
  name: jwt-production
plugin: jwt
config:
  key_claim_name: kid
  claims_to_verify:
    - exp
  maximum_clock_skew: 300

# 3. 增加Token过期缓冲时间
# Token有效期设为2小时，但实际使用1.5小时
# 避免临界时间问题
```

**经验教训**:
1. 所有服务器必须配置NTP同步
2. JWT验证要考虑时钟偏差
3. 监控时钟偏差指标
4. Token有效期要留足缓冲时间

## 踩坑记录

### Q1: Kong插件配置不生效
**原因**: 插件未在KongAdmin端口正确加载
**解决**: 检查Kong Pod日志，确认插件已启用

### Q2: Istio sidecar注入后Pod启动慢
**原因**: istio-proxy容器初始化耗时
**解决**: 配置holdApplicationUntilProxyStarts: true

### Q3: mTLS导致服务间通信失败
**原因**: 部分服务未启用sidecar
**解决**: 使用PeerAuthentication设置PERMISSIVE模式过渡

### Q4: 熔断器误触发
**原因**: 熔断阈值设置过低
**解决**: 调整consecutiveErrors和interval参数

### Q5: Jaeger链路追踪数据不完整
**原因**: 未配置注入注解
**解决**: 在Deployment中添加sidecar.istio.io/inject: "true"
