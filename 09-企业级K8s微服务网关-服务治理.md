## 六、网关方案对比与选型

### 6.1 主流网关对比

| 特性 | Kong | Apache APISIX | Nginx Ingress | Envoy |
|------|------|--------------|---------------|-------|
| 基于 | OpenResty | OpenResty | Nginx | C++ |
| 配置方式 | Admin API + DB | etcd + Admin API | K8s CRD | xDS API |
| 插件生态 | 200+ | 80+ | 50+ | 30+ |
| 性能 | 极高 | 极高 | 高 | 高 |
| 动态配置 | ✅ | ✅ | ❌ 需reload | ✅ |
| 热更新 | ✅ | ✅ | ❌ | ✅ |
| 服务发现 | ✅ DNS | ✅ DNS/etcd | ✅ K8s | ✅ xDS |
| 限流 | ✅ 内置 | ✅ 内置 | ✅ 内置 | ✅ 内置 |
| 认证 | ✅ JWT/OAuth2 | ✅ JWT/key-auth | ✅ OAuth2 | ✅ JWT |
| 运维界面 | ✅ Kong Manager | ✅ Dashboard | ❌ | ❌ |
| 学习曲线 | ⭐⭐ | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| 适用场景 | 通用API网关 | 高性能网关 | K8s Ingress | Service Mesh |

**选型建议**:
- K8s环境 + 需要Ingress → Nginx Ingress Controller
- 需要丰富插件生态 → Kong 或 APISIX
- 微服务架构 + Service Mesh → Envoy (Istio数据面)
- 本项目选择: Nginx Ingress(入口) + Istio(服务间)

### 6.2 服务网格 vs 传统网关

| 特性 | 传统网关(Nginx/Kong) | Service Mesh(Istio/Envoy) |
|------|---------------------|--------------------------|
| 部署方式 | 集中部署 | Sidecar每Pod一个 |
| 管理面 | 网关配置 | 控制面+数据面 |
| 流量管理 | 网关层 | 应用层(7层) |
| 可观测性 | 网关日志 | 全链路追踪+指标 |
| 安全 | TLS终结 | mTLS双向认证 |
| 灰度发布 | 网关路由 | 流量权重分配 |
| 故障注入 | ❌ | ✅ |
| 资源开销 | 低 | 中(每Pod增加~50MB) |
| 适用规模 | 100+服务 | 500+服务 |

## 九、真实故障案例

### 案例1: Istio Sidecar导致Pod启动延迟30秒

**故障**: 应用Pod启动后30秒才Ready，影响滚动更新速度

**原因**: Istio sidecar默认等待应用容器完全启动后才注入

**解决**:
```yaml
# 在Pod annotation中添加
metadata:
  annotations:
    proxy.istio.io/config: '{"holdApplicationUntilProxyStarts": true}'
    
# 或者在MeshConfig中全局配置
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    defaultConfig:
      holdApplicationUntilProxyStarts: true
```

### 案例2: 服务网格延迟增加5ms

**故障**: 启用Istio后P99延迟从15ms增加到20ms

**原因**: Sidecar代理增加了网络跳数，遥测采集增加了开销

**解决**:
```yaml
# 1. 降低采样率
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: reduce-sampling
spec:
  tracing:
    - randomSamplingPercentage: 1  # 从10%降到1%

# 2. 禁用不必要的访问日志
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: disable-access-log
spec:
  accessLogging:
    - providers: []  # 禁用

# 3. 使用headless service减少DNS查询
```

### 案例3: Kong限流在Redis集群下失效

**故障**: Kong rate-limiting插件配置了Redis集群，但限流不生效

**原因**: Kong默认使用Redis单机模式连接，不支持Redis Cluster

**解决**:
```yaml
# 方案1: 使用Redis Sentinel
plugins:
  - name: rate-limiting
    config:
      minute: 100
      policy: redis
      redis_host: redis-sentinel
      redis_port: 26379
      redis_sentinel_master: mymaster

# 方案2: 使用外部Redis单机(推荐)
# 部署独立的Redis实例给Kong使用
```

## 十、性能压测

### 10.1 k6压测脚本

```javascript
// load-test.js - k6负载测试
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '2m', target: 100 },   // 爬坡
    { duration: '5m', target: 100 },   // 持续负载
    { duration: '2m', target: 500 },   // 峰值
    { duration: '5m', target: 500 },   // 峰值持续
    { duration: '2m', target: 0 },     // 缓降
  ],
  thresholds: {
    http_req_duration: ['p(99)<200'],  // P99 < 200ms
    http_req_failed: ['rate<0.01'],    // 错误率 < 1%
  },
};

const BASE_URL = 'http://istio-ingressgateway:80';

export default function () {
  // 用户列表API
  const usersRes = http.get(`${BASE_URL}/api/v1/users?page=1&size=20`, {
    headers: { Authorization: `Bearer ${__ENV.TOKEN}` },
  });
  check(usersRes, {
    'users status 200': (r) => r.status === 200,
    'users latency < 100ms': (r) => r.timings.duration < 100,
  });

  sleep(1);

  // 创建订单API
  const orderRes = http.post(`${BASE_URL}/api/v1/orders`,
    JSON.stringify({ product_id: 1, quantity: 1 }),
    { headers: { 'Content-Type': 'application/json', Authorization: `Bearer ${__ENV.TOKEN}` } }
  );
  check(orderRes, {
    'order status 201': (r) => r.status === 201,
    'order latency < 200ms': (r) => r.timings.duration < 200,
  });

  sleep(Math.random() * 3 + 1);
}
```

NAMESPACE=${2:-production}

echo "========== 金丝雀发布: ${SERVICE} =========="

echo "Step 1: 部署canary版本..."
kubectl apply -f - << EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ${SERVICE}-canary
  namespace: ${NAMESPACE}
spec:
  replicas: 1
  selector:
    matchLabels:
      app: ${SERVICE}
      version: canary
  template:
    metadata:
      labels:
        app: ${SERVICE}
        version: canary
    spec:
      containers:
        - name: ${SERVICE}
          image: harbor.internal.com/production/${SERVICE}:canary
EOF

echo "Step 2: 配置流量分配(${CANARY_WEIGHT}%到canary)..."
kubectl apply -f - << EOF
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: ${SERVICE}
  namespace: ${NAMESPACE}
spec:
  hosts:
    - ${SERVICE}
  http:
    - route:
        - destination:
            host: ${SERVICE}
            subset: stable
          weight: $((100 - CANARY_WEIGHT))
        - destination:
            host: ${SERVICE}
            subset: canary
          weight: ${CANARY_WEIGHT}
EOF

echo "Step 3: 监控canary指标(5分钟)..."
echo "  查看Grafana dashboard: http://grafana.internal.com/d/canary"
echo "  关注: 错误率、延迟、QPS"

echo ""
echo "Step 4: 确认后全量切换?"
read -p "输入yes继续: " CONFIRM
if [ "${CONFIRM}" = "yes" ]; then
  kubectl set image deployment/${SERVICE}     ${SERVICE}=harbor.internal.com/production/${SERVICE}:canary     -n ${NAMESPACE}
  kubectl rollout status deployment/${SERVICE} -n ${NAMESPACE} --timeout=300s
  echo "✅ 全量发布完成"
else
  echo "取消发布，回滚canary..."
  kubectl delete deployment ${SERVICE}-canary -n ${NAMESPACE}
  echo "✅ 已回滚"
fi
```

### 11.2 紧急预案

```
场景1: 网关502/504
  1. 检查后端服务Pod状态
  2. 检查Istio sidecar状态
  3. 查看Envoy access log
  4. 临时方案: 重启受影响的Pod

场景2: 服务间调用超时
  1. 检查VirtualService timeout配置
  2. 检查DestinationRule circuitBreaker
  3. 检查目标服务日志
  4. 临时方案: 增加超时时间或禁用熔断

场景3: mTLS证书过期
  1. 检查istiod证书
  2. 手动触发证书轮换
  3. 重启受影响的Pod
  4. 检查Citadel组件状态
```

## 十二、容量规划

| 指标 | 单Istio代理 | 推荐配置 |
|------|------------|---------|
| CPU | 0.1核 | 每Pod预留0.1核 |
| 内存 | 50MB | 每Pod预留128MB |
| 延迟开销 | 1-5ms | 可接受范围 |
| 最大QPS | 10,000+ | 单代理 |

**集群规模建议**:
- 100个服务以下 → 可以不用Service Mesh
- 100-500个服务 → Istio轻量级部署
- 500+服务 → Istio生产级部署+高可用
