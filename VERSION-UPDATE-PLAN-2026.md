## 二、更新优先级

### P0 - 立即更新（合规风险）

1. **MySQL 8.0 → 8.4 LTS**
   - 原因：8.0已于2026-04-30 EOL，等保审计直接失败
   - 影响：认证插件变化、SQL Mode变化
   - 回滚方案：保留8.0数据目录，可回滚

2. **Kubernetes 1.31 → 1.35.4**
   - 原因：1.31已于2025-10 EOL，存在已知漏洞
   - 影响：API变化、特性变化
   - 回滚方案：etcd快照备份

3. **Harbor 2.12 → 2.15.1**
   - 原因：2.12已于2026-03 EOL，镜像仓库安全风险
   - 影响：数据库迁移、配置变化
   - 回滚方案：保留旧版本数据目录

### P1 - 计划更新（技术债务）

4. **Redis 7.2 → 8.6.3**
   - 原因：7.2 Active Support已结束
   - 影响：新架构变化
   - 回滚方案：RDB备份

5. **Prometheus 2.50 → 3.11.3**
   - 原因：2.x已过时
   - 影响：配置格式变化
   - 回滚方案：保留旧配置

6. **Thanos 0.34 → 0.41.0**
   - 原因：0.34已过时
   - 影响：组件变化
   - 回滚方案：保留旧配置

### P2 - 优化更新（性能提升）

7. **Calico 3.28 → 3.32.0**
   - 原因：3.28已过时
   - 影响：eBPF支持改进
   - 回滚方案：保留旧配置

8. **Kong 3.5 → 3.9.1**
   - 原因：3.5已过时
   - 影响：插件变化
   - 回滚方案：保留旧配置

9. **Istio 1.20 → 1.29.2**
   - 原因：1.20早已EOL
   - 影响：Ambient Mesh支持
   - 回滚方案：保留旧配置

## 四、回滚计划

### 通用回滚步骤

1. **备份当前状态**
   - etcd快照备份
   - 数据库备份
   - 配置文件备份

2. **停止服务**
   - 停止新版本服务
   - 恢复旧版本配置

3. **启动旧版本**
   - 启动旧版本服务
   - 验证服务状态

4. **验证回滚**
   - 检查日志
   - 验证功能
   - 确认数据完整性

### 组件特定回滚

| 组件 | 回滚方式 | 数据丢失风险 |
|------|----------|--------------|
| Kubernetes | etcd快照恢复 | 低(快照后数据) |
| MySQL | 保留8.0数据目录 | 低(可回滚) |
| Redis | RDB恢复 | 低(AOF everysec) |
| Harbor | 保留旧版本数据目录 | 低(可回滚) |
| Prometheus | 保留旧配置 | 无(指标数据) |
